Datenmodell

Wie die Daten in PostgreSQL strukturiert werden

Entity-Relationship Diagramm
User
id UUID (PK)
email String
name String
password String (hashed)
role TEACHER | STUDENT
createdAt DateTime
Workspace
id UUID (PK)
name String (z.B. "Analysis")
description String?
ownerId UUID → User (Lehrer)
createdAt DateTime
updatedAt DateTime
WorkspaceMember
id UUID (PK)
workspaceId UUID → Workspace
userId UUID → User (Schüler)
joinedAt DateTime
Verknüpfung: Welcher Schüler hat Zugriff auf welchen Workspace
Page
id UUID (PK)
workspaceId UUID → Workspace
title String? (z.B. "Aufgabe 3")
sortOrder Int
backgroundType BLANK | GRID | LINED | COORDINATE
canvasState JSONB (tldraw Snapshot)
createdAt DateTime
updatedAt DateTime
FileUpload
id UUID (PK)
pageId UUID → Page
uploadedBy UUID → User
filename String
mimeType String
storagePath String
createdAt DateTime
Beziehungen
User (Lehrer) → besitzt → n Workspaces
User (Schüler) → Mitglied in → n Workspaces
Workspace → enthält → n Pages
Page → hat → n FileUploads
Page → speichert → 1 canvasState (JSON)

Beispiel-Szenario

Du (Lehrer) erstellst Workspace "Trigonometrie" → weist deinem Sohn zu
→ fügst 3 Pages hinzu: "Sinussatz" (kariert), "Kosinussatz" (kariert), "Übungen" (Koordinatensystem)
→ lädst ein PDF mit Aufgaben auf Page "Übungen" hoch
→ dein Sohn öffnet den Workspace, sieht die 3 Seiten, löst Aufgaben mit Stift
→ du siehst seine Arbeit live oder später, korrigierst mit rotem Stift
